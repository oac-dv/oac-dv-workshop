Exercise 1 · The foundation

Why a dataset is essential

A dataset is a self-service data model — one or more tables with the relationships between them defined. Individually, each table tells only part of the story. The dataset is what stitches them into one analytical model we can actually ask questions of.

Raw columns don't answer business questions. Metrics do.

Policies, premiums, claims, drivers and vehicles each hold one slice of the customer. Until they're connected, you can't compute churn rate, loss ratio or customer value — those numbers live across tables, not inside any one of them. The dataset is the trusted layer all three canvases share, so everyone is working from the same definitions.

Exercise 1 · Concept

Joins: how separate tables become one trusted dataset

A join tells OAC how rows from two tables belong together. In our workshop, the common key was usually customer_policy_id: when the policy row and claim row have the same ID, OAC treats them as the same customer story instead of two unrelated lists.

What is a join?

Think of a join as a matching rule plus a keep rule. The matching rule says which values must line up. The keep rule says what happens when a row has no match.

Inner / equi join: keep only rows where the keys are equal on both sides. In SQL terms, the equality part is the equi join condition: Policy ID = Policy ID.
Key distinction: equi describes the match condition, usually key equals key. left, right, and inner describe which unmatched rows stay in the result.

Why we used left joins in the workshop

A left join keeps all rows from the left, or driving, table even when there is no match on the right. We made Policies the driving table and left-joined the related tables to it. That choice matters: a customer who never filed a claim still belongs in the retention analysis.

The logic: retention is about customers, not only claims. Left-joining onto Policies guarantees no customer disappears just because a related table has no row for them.

The concept that trips everyone up

Preserve grain: stop one customer from becoming three customers

Grain means what one row represents. Policies is one row per customer policy. Claims is one row per claim. When one policy has three claims, the join creates three joined rows. That is normal. The risk is accidentally adding the same policy-level measure three times.

Same data, same join — the only difference is Preserve Grain on the Policies table. It holds the analysis at the policy grain so a one-to-many join cannot multiply policy-level measures.

How to explain it to a new person

A join can legitimately create more rows because it brings together details from different levels. That is not automatically wrong. The problem happens when a value from the higher-level table, like premium, is repeated across those lower-level rows and then summed as if each repeat were a new premium.

Short version: joins answer, "Which rows belong together?" Grain answers, "What does one row mean?" Preserve Grain protects measures when those two ideas collide.
The tell-tale symptom: if customer count or premium total is mysteriously higher than reality, suspect fan-out first — a join is multiplying rows, and grain plus COUNT(DISTINCT) is the cure.
Exercise 1 · Before you save

IDs are labels, not numbers

customer_policy_id, Claim ID, Driver ID all look numeric, but they identify things — they're attributes, not measures. Before saving the dataset we switched every ID column from Measure (#) to Attribute (A). Skip this and OAC tries to add IDs together, and your counts go haywire.

✗ Left as a measure

OAC sums the id

SUM(customer_policy_id)

Adding id 1001 + 1002 + 1003 produces a meaningless giant number — and any "count" built on it is wrong.

✓ Set as an attribute

OAC counts customers

COUNT(DISTINCT customer_policy_id)

Treated as a label, the id becomes something you group and count by — the real size of the customer base.

Exercise 2 · Business language

The six decision metrics

Each calculation turns raw fields into a number the business actually asks for. We understood the "why" before typing the formula.

01

Policy Count

How many unique customers do we have? DISTINCT collapses fanned-out rows back to one per customer.

COUNT(DISTINCT customer_policy_id)
02

Claims Count

How many claim events happened — incidents, not dollars.

COUNT(DISTINCT "Claim ID")
03

Churned Policy Count

How many customers already left? FILTER counts only churned rows, ignoring the rest.

FILTER(COUNT("End of Policy Renewal") USING "Renewal Status"='churned')
04

Churn Rate

What share of customers churned? 50 churns means different things in a base of 200 vs 2,000.

"Churned Policy Count" / "Policy Count"
05

Loss Ratio

For every $1 of premium, how much went back out in claims? 0.60 = 60¢ paid per dollar earned.

"Insurance Paid" / "Premium Amount"
06

Customer Value Score

Net value per customer = premium collected minus claims paid. The X-axis of the retention scatter.

"Premium Amount" - "Insurance Paid"

What makes an AI Agent different — its four parts

An Assistant builds a chart. An Agent applies your organisation's rules and vocabulary before it answers — that's the governance layer.

1
Dataset — the facts

One indexed dataset the agent is grounded in. Column scope and filters constrain what it can see.

2
Supplemental instructions — the brain

Up to ~6,000 chars of your terminology, KPI formulas and response format. Without them the agent guesses.

3
Knowledge documents — the library

PDF/TXT files retrieved via RAG, adding policy and process context beyond the columns.

4

Onboarding text and sample prompts so users know what the agent can answer.

Readiness work behind the scenes

After seeing the Homepage Assistant, Workbook Assistant, and AI Agent in action, we closed with the pre-work required to make those responses reliable. An admin, power user, or dataset domain expert prepares the dataset by indexing it, adding useful synonyms, certifying trusted sources, setting column scope, and checking data types. Synonyms are business-language mappings to database columns and values, so a user can ask for "premium", "claims paid", or "customers" and OAC knows which fields those words mean.

Why indexing matters

Indexing makes a dataset searchable for the AI Assistant, Homepage Search, and Agents. An un-indexed dataset may be invisible to AI, or the AI may not know which columns and values can answer the prompt.
